# แผน Cut-over — สลับ Maru Waffle ไปใช้ Supabase 100%

อัปเดต: เตรียมเปิดใช้งานจริง · ย้ายข้อมูลครบ + เปิดแจ้งเตือน LINE ฝั่ง Supabase

---

## ภาพรวม
ตอนนี้แอป (หน้าเว็บ/PWA) เขียน-อ่านข้อมูลผ่าน **Supabase** หมดแล้ว เหลือ 3 อย่างที่ต้องทำให้ครบก่อนเลิกใช้ Google Sheet + Apps Script:
1. ย้ายข้อมูลเก่าที่เหลือเข้ามาให้ครบ (สคริปต์ `migrateAllToSupabase`)
2. เปิดแจ้งเตือน LINE ฝั่ง Supabase (Edge Function `notifyLine`)
3. สลับให้ทุกคนใช้แอปตัวใหม่ + หยุดระบบเดิม

---

## ส่วนที่ 1 — เตรียมก่อนวัน Cut-over (ทำล่วงหน้าได้)

### 1.1 ตั้งค่า LINE ใน Supabase (ครั้งเดียว)
ไปที่ **Supabase → Edge Functions → Manage secrets** เพิ่ม 2 ค่า:
- `LINE_CHANNEL_TOKEN` = Channel access token (เอาจาก LINE Developers → Messaging API channel ของร้าน — เป็นตัวเดียวกับที่อยู่ใน Code.gs เดิม บรรทัด `LINE_CHANNEL_TOKEN`)
- `LINE_GROUP_ID` = id กลุ่มไลน์ที่จะส่งแจ้งเตือน (ตัวเดียวกับ `LINE_GROUP_ID` ใน Code.gs เดิม)

> มีอยู่แล้วใน Code.gs: token ขึ้นต้น `RPcMHkv...` และ group id `C655668...`

### 1.2 Deploy Edge Function ตัวใหม่
อัปโหลด/Deploy `secure-api_index.ts` (เวอร์ชันใหม่ที่มี action `notifyLine`)
- Deploy ผ่าน Supabase Dashboard → Edge Functions → secure-api → วางโค้ดใหม่ → Deploy
- **ปิด "Verify JWT"** เหมือนเดิม

### 1.3 อัปโหลดไฟล์แอปขึ้น GitHub
ไฟล์ที่แก้รอบนี้: `shared.js`, `sb-data.js`, `sw.js` (เวอร์ชัน v55)
- อัปโหลดทับใน repo `Maru-Waffle-V2` → GitHub Pages จะอัปเดตเอง
- `sw.js` บั๊มเป็น v55 แล้ว เครื่องพนักงานจะดึงโค้ดใหม่อัตโนมัติ (อาจต้องปิด-เปิดแอป 1 ครั้ง)

### 1.4 ทดสอบ LINE (ก่อนวันจริง)
- เปิดแอป → บันทึกรายงานสิ้นวันของ "วันทดสอบ" 1 ครั้ง
- ดูว่าการ์ดสรุปยอดเข้า LINE กลุ่มถูกต้อง (หน้าตาเหมือนเดิม)
- ถ้าไม่เข้า: เช็ค secrets 2 ตัว + ว่า Deploy Edge ใหม่แล้ว

## ส่วนที่ 2 — วัน Cut-over (ทำตามลำดับ)

### 2.1 ให้พนักงานกรอกข้อมูล "วันสุดท้าย" ผ่านระบบเดิมให้จบ
- รอจนปิดร้าน/กรอกครบ (ประมาณ 21:00)

### 2.2 ย้ายข้อมูลรอบสุดท้าย
- เปิด Google Sheet → ส่วนขยาย → Apps Script
- กดรัน `migrateAllToSupabase` อีกครั้ง (ค่า `MIGRATE_FROM` เดิม)
- ดู Logs: ตาราง `sales`, `stock_*`, `attendance` ต้องขึ้น "ส่งใหม่ X แถว" = ข้อมูลวันนี้เข้าครบ
- สคริปต์กันซ้ำอยู่แล้ว รันกี่ครั้งก็ไม่เพิ่มของซ้ำ

### 2.3 ตรวจสอบข้อมูลใน Supabase (Verify)
เปิด Supabase → Table Editor เช็คอย่างน้อย:
- `sales` — มีแถวของวันล่าสุด ยอดถูกต้อง
- `stock_withdraw` / `stock_receive` — รายการล่าสุดมาครบ
- `attendance` — มีบันทึกเข้า-ออกล่าสุด
- จำนวนแถวรวมใกล้เคียงกับในชีท (เทียบช่วงวันที่ที่ย้าย)

### 2.4 ประกาศสลับระบบ
- แจ้งพนักงานทุกคน: **พรุ่งนี้เริ่มใช้แอปตัวใหม่ (ลิงก์ GitHub Pages) เท่านั้น — หยุดกรอกใน Google ฟอร์ม/ระบบเดิม**
- ปักลิงก์แอปไว้ในกลุ่มไลน์ + ติดตั้งลงหน้าจอ (Add to Home Screen)

## ส่วนที่ 3 — หลัง Cut-over

### 3.1 ตัดการพึ่งพา Apps Script (ไม่บังคับ ทำเมื่อมั่นใจ)
- ในแอปยังมี `APPS_SCRIPT_URL` เป็น fallback แต่ไม่มี action ไหนวิ่งไปแล้ว
- เมื่อใช้ Supabase ได้ราบรื่น 2–3 วัน ค่อยลบ fallback ออกได้ (ไม่ลบก็ไม่กระทบ)

### 3.2 หมุน key ที่เคยหลุด (สำคัญด้านความปลอดภัย)
- `service_role key` (ที่ใส่ใน Apps Script ตอนย้ายข้อมูล) → หลังย้ายเสร็จ **ลบออกจาก Code.gs** แล้ว rotate ใหม่ใน Supabase → Project Settings → API
- ตรวจว่า key ใหม่ไม่ได้ถูกฮาร์ดโค้ดในไฟล์ที่ push ขึ้น GitHub (แอปใช้ publishable key เท่านั้น ซึ่งปลอดภัยอยู่แล้ว)

### 3.3 เก็บ Google Sheet เป็น backup
- อย่าเพิ่งลบชีท เก็บไว้เป็นข้อมูลสำรองอย่างน้อย 1–2 เดือน
- ปิดสิทธิ์แก้ไขให้พนักงาน (กันกรอกผิดที่)

---

## แผนถอย (Rollback) — ถ้ามีปัญหาวันแรก
1. ระบบเดิม (Google Sheet + Apps Script) **ยังอยู่ครบ ไม่ได้ลบ** → ให้พนักงานกลับไปใช้ชั่วคราวได้ทันที
2. ข้อมูลที่กรอกในแอปใหม่ (Supabase) ช่วงที่มีปัญหา ยังอยู่ใน Supabase → ดึงกลับ/ย้ายเข้าชีทภายหลังได้
3. แจ้งให้แก้ปัญหาเฉพาะจุด แล้วค่อยสลับใหม่อีกครั้ง

## สถานะแจ้งเตือน LINE (รอบนี้)
| แจ้งเตือน | สถานะ |
|---|---|
| ✅ สรุปยอดสิ้นวัน (รายงานสิ้นวัน) | **เปิดใช้แล้ว** (Edge `notifyLine` + Flex การ์ดเหมือนเดิม) |
| ⏳ ปิดร้าน (สรุปสต๊อก) | รอเพิ่ม (งานต่อเนื่อง) |
| ⏳ ออดิทสต๊อก | รอเพิ่ม (งานต่อเนื่อง) |
| ⏳ เข้า-ออกงาน | รอเพิ่ม (งานต่อเนื่อง) |

> อีก 3 ตัวเป็นการคัดลอก Flex builder แบบเดียวกับรายงานสิ้นวัน — ทำต่อให้ครบได้ก่อนหรือหลังวัน cut-over ก็ได้ (รายงานสิ้นวันคือตัวสำคัญสุดที่ใช้ทุกวัน)
